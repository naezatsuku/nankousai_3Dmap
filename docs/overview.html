第3章 セキュリティ設計

⚠ 本章は許可申請の審査において特に重要な箇所です。 セキュリティ上の懸念点に対する具体的な対策を記載しています。

3.1 認証システム

採用技術

本サービスは Supabase Auth（業界標準の認証基盤・ISO 27001 取得済みのインフラ上で動作）を採用しています。

ログインの仕組み

1ユーザーがメールアドレス・パスワードを入力
↓
2Supabase が認証を行い JWT（JSON Web Token） を発行
↓
3JWT はブラウザの Cookie に保存（HttpOnly・Secure フラグ付き）
→ JavaScript から直接読み取れないため XSS 攻撃への耐性が高い
↓
4以後のすべてのリクエストに JWT が自動付与される
↓
5サーバーサイドで JWT を検証してから処理を実行

パスワードの保護

  • パスワードは bcrypt でハッシュ化して保存（平文では一切保存しない）
  • 最低8文字を必須とし、強度インジケーターで安全なパスワードを誘導
  • パスワードリセットはメールリンク経由のみ（管理者であっても他人のパスワードは不明）

セッション管理

  • JWT の有効期限：1時間（期限切れ前に自動更新）
  • ログアウト時は JWT を無効化しセッションを完全破棄
  • 不正な JWT でのアクセスはすべてサーバーで拒否

3.2 なりすまし防止

ロールベースアクセス制御（RBAC）

ユーザーのロール（admin / editor / student）はサーバーサイドのデータベースに保存されています。

なりすましが不可能な理由

✗ ロール情報をブラウザの Cookie や localStorage に保存しない
✓ API アクセス時に毎回サーバーで JWT の user_id からロールを照合する

例： editor が admin ページにアクセスしようとした場合
→ サーバーが JWT から user_id を取得
→ profiles テーブルで role = 'editor' を確認
→ 403 Forbidden を返却（アクセス不可）

担当クラスの制限

editor は exhibit_editors テーブルで許可された展示 ID のみ編集可能です。他クラスの情報を改ざんすることは技術的に不可能です。

例：高1-1 の editor が 高2-3 の情報を書き換えようとした場合
  → API が JWT の user_id を確認
  → exhibit_editors テーブルに「高2-3 への許可」がないと判定
  → 403 Forbidden を返却（操作不可）

行レベルセキュリティ（RLS）

PostgreSQL の RLS（Row Level Security） により、データベース層でもアクセス制御を実施しています。アプリケーション層にバグがあってもデータベース層でブロックします。

-- 例：シフト表は自分のクラスのメンバーのみ参照可能
CREATE POLICY "self read on student_exhibits"
  ON student_exhibits FOR SELECT TO authenticated
  USING (user_id = auth.uid());

-- 例：シフト割り当ては service_role のみ書き込み可能
-- → 認証済みユーザーは読み取りのみ、書き込みはサーバーのAPI経由

3.3 スタンプラリーの不正防止

QR コードの仕組み（HMAC-SHA256 署名）

QR URL の構造

/stamp?e={展示ID}&w={時間ウィンドウ}&h={HMAC署名(16文字)}

時間ウィンドウ = Unix時刻 ÷ 60秒 の整数 → 1分ごとに変化

検証フロー

1来場者が QR を読み取り、サーバーに送信
↓
2サーバーが展示の stamp_secret（DB 上の秘密鍵）で HMAC を再計算
↓
3QR の署名と一致すれば正規 QR と判定。±1ウィンドウ（最大2分）を許容
↓
4スタンプを付与（同一展示は UNIQUE 制約で重複防止）

不正が不可能な理由

攻撃手法対策
QR 画像の写真を転用1分ごとに QR が変化するため最大2分後に無効化
QR URL を自作・改ざんstamp_secret（サーバーのみ保有の秘密鍵）なしに正規の署名を生成することは計算上不可能
同じ展示を複数回スキャンstamps テーブルの UNIQUE 制約（user_id × exhibit_id）で重複を防止
他人の user_id を使用JWT 認証により本人以外の user_id でのアクセス不可

3.4 データ・著作権保護

画像の保護

  • 全 <img> タグに pointer-events: none + -webkit-user-drag: none を設定（ドラッグ保存防止）
  • 右クリックメニューを JavaScript でブロック
  • テキスト選択禁止（入力欄は除く）

サービスロールキーの保護

  • Supabase の service_role_key（管理者権限キー）は環境変数（.env.local）にのみ保存
  • フロントエンドのコードには一切含まれない
  • サーバーサイドの API Route からのみアクセス可能
  • Vercel のシークレット管理機能で本番環境に設定

プッシュ通知のセキュリティ

  • FCM トークンはユーザーの user_id に紐付けて管理
  • 通知の送信は service_role_key を持つ API のみが実行可能
  • ユーザーは自分の通知設定のみ変更可能（他人の通知操作は不可）

第5章 生徒（student）向けガイド

5.1 アカウント作成

1管理画面（/admin）→「アカウント作成」
↓
2メールアドレス・パスワード（8文字以上）を入力して登録
↓
3初回ログイン時にプロフィール（名前・学年・クラス・出席番号）を設定
↓
4担当の先生（editor/admin）にクラスへの割り当てを依頼する
⚠ クラスに割り当てられるまでシフト機能は使用できません。

5.2 シフトアンケート

管理画面 → サイドバー「シフト管理」→「シフトアンケート」

選択肢意味
◎ 入りたいこの時間帯に入りたい（優先的に割り当てられる）
△ どちらでもいいどちらでも対応可能（◎が不足した場合に補填）
✕ 入れないこの時間帯には入れない（絶対に割り当てられない）
⚠ ✕ を選択したコマには絶対にシフトが割り当てられません。確実に都合が悪い場合のみ使用してください。

回答に締め切りはなく、割り当て確定前であれば何度でも変更できます。

5.3 シフト表の確認とシフト通知

管理画面 → 「シフト表」→ 土/日ボタンで日付切替。自分のコマはオレンジ色で表示。

シフト表右上の 🔔 ボタンから「何分前に通知するか」を設定すると、「予定」ページを開いたときに通知がスケジュールされます。

第6章 担当者（editor）向けガイド

6.1 ログインと担当展示

管理画面（/admin）にアクセス → メールアドレス・パスワードを入力。

⚠ editor は admin または別の editor に「担当展示の割り当て」をしてもらう必要があります。割り当てがない場合は展示情報を編集できません。

6.2 展示情報の編集

管理画面 → 「展示編集」→ 担当展示をクリック。

タブ編集できる内容
① 基本情報展示名・キャッチコピー・説明文・表紙画像・場所・フロア
② コンテンツセクション（見出し＋本文＋画像/動画）の追加・並び替え。フードの場合はメニュー管理（名前・価格・在庫・販売数）
③ 設定・クイック待ち時間更新・スタンプラリー参加設定

6.3 お知らせの投稿

管理画面 → 「お知らせ管理」→「新規作成」。タイトル・本文・画像・重要マークを設定して「投稿する」。投稿と同時にフォローしているユーザーへプッシュ通知が送信されます。

6.4 かんたん更新（当日使用）

展示編集ページの「かんたん更新」ボタンから当日の素早い更新が可能。

  • 待ち時間の更新（待ち組数 × 1組あたりの時間）
  • スタンプ QR の表示
  • フードメニューの在庫・販売数・販売状況
  • コメントの承認・削除

6.5 シフト管理

① メンバー登録

管理画面 → 「メンバー管理」→ 学年・クラスでフィルタリングして生徒を割り当て。「表示中を一括追加」ボタンで一括登録が可能。

② 時間設定・コマ生成

管理画面 → 「シフト編集」→「時間設定」タブ。開始/終了時刻・コマの長さ・必要人数を設定して「コマを生成する」。

③ アンケート確認

「アンケート確認」タブで生徒全員の希望（◎△✕）をマトリクスで確認。

④ シフト割り当て

「シフト割り当て」タブ。

方法説明
🤖 自動割当グリーディ法で自動割り当て。◎優先→△補填、✕は絶対割り当てなし。結果に警告・過負荷者を表示。
手動割当＋ボタンから生徒をピッカーで選択。アンケート回答と既存割り当てコマ数を参考に選択可能。

8.2 来場者の匿名性

ログイン不要のユーザーは、端末のローカルストレージに保存されたランダムな UUID（例：a1b2c3d4-e5f6-...）のみで識別されます。

  • 氏名・メールアドレス等の個人を特定できる情報は収集しない
  • スタンプ・予定・投票記録はこの UUID に紐付き、本人しか閲覧できない
  • UUID は端末を変更すると引き継がれない（意図的な設計）

8.3 データの保管と管理

  • すべてのデータは Supabase（アジア太平洋リージョン）に保管
  • パスワードは bcrypt でハッシュ化して保存（平文では保管しない）
  • 通信はすべて HTTPS（TLS 1.3） で暗号化
  • ログインユーザーの情報は当事者以外の利用者からは閲覧不可

8.4 データの保存期間

文化祭終了後のデータ保持期間・削除方針については学校の情報管理規定に従います。ユーザーは admin にアカウント削除をリクエストできます。

8.5 第三者提供

収集した個人情報は以下の場合を除き、第三者に提供しません。

  • 法令に基づく場合
  • 人の生命・身体または財産の保護のために必要な場合
